5 ## Introduction
6 
7 Now that you have read the [googletest Primer](primer.md) and learned how to
8 write tests using googletest, it's time to learn some new tricks. This document
9 will show you more assertions as well as how to construct complex failure
10 messages, propagate fatal failures, reuse and speed up your test fixtures, and
11 use various flags with your tests.

13 ## More Assertions
14 
15 This section covers some less frequently used, but still significant,
16 assertions.
17 
18 ### Explicit Success and Failure
19 
20 These three assertions do not actually test a value or expression. Instead, they
21 generate a success or failure directly. Like the macros that actually perform a
22 test, you may stream a custom failure message into them.
23 
24 ```c++
25 SUCCEED();
26 ```
27 
28 Generates a success. This does **NOT** make the overall test succeed. A test is
29 considered successful only if none of its assertions fail during its execution.
30 
31 NOTE: `SUCCEED()` is purely documentary and currently doesn't generate any
32 user-visible output. However, we may add `SUCCEED()` messages to googletest's
33 output in the future.
34 
35 ```c++
36 FAIL();
37 ADD_FAILURE();
38 ADD_FAILURE_AT("file_path", line_number);
39 ```
40 
41 `FAIL()` generates a fatal failure, while `ADD_FAILURE()` and `ADD_FAILURE_AT()`
42 generate a nonfatal failure. These are useful when control flow, rather than a
43 Boolean expression, determines the test's success or failure. For example, you
44 might want to write something like:
45 
46 ```c++
47 switch(expression) {
48  case 1:
49  ... some checks ...
50  case 2:
51  ... some other checks ...
52  default:
53  FAIL() << "We shouldn't get here.";
54 }
55 ```
56 
57 NOTE: you can only use `FAIL()` in functions that return `void`. See the
58 [Assertion Placement section](#assertion-placement) for more information.
59 
60 ### Exception Assertions
61 
62 These are for verifying that a piece of code throws (or does not throw) an
63 exception of the given type:
64 
65 Fatal assertion | Nonfatal assertion | Verifies
66 ------------------------------------------ | ------------------------------------------ | --------
67 `ASSERT_THROW(statement, exception_type);` | `EXPECT_THROW(statement, exception_type);` | `statement` throws an exception of the given type
68 `ASSERT_ANY_THROW(statement);` | `EXPECT_ANY_THROW(statement);` | `statement` throws an exception of any type
69 `ASSERT_NO_THROW(statement);` | `EXPECT_NO_THROW(statement);` | `statement` doesn't throw any exception
70 
71 Examples:
72 
73 ```c++
74 ASSERT_THROW(Foo(5), bar_exception);
75 
76 EXPECT_NO_THROW({
77  int n = 5;
78  Bar(&n);
79 });
80 ```
81 
82 **Availability**: requires exceptions to be enabled in the build environment

84 ### Predicate Assertions for Better Error Messages
85 
86 Even though googletest has a rich set of assertions, they can never be complete,
87 as it's impossible (nor a good idea) to anticipate all scenarios a user might
88 run into. Therefore, sometimes a user has to use `EXPECT_TRUE()` to check a
89 complex expression, for lack of a better macro. This has the problem of not
90 showing you the values of the parts of the expression, making it hard to
91 understand what went wrong. As a workaround, some users choose to construct the
92 failure message by themselves, streaming it into `EXPECT_TRUE()`. However, this
93 is awkward especially when the expression has side-effects or is expensive to
94 evaluate.
95 
96 googletest gives you three different options to solve this problem:
97 
98 #### Using an Existing Boolean Function
99 
100 If you already have a function or functor that returns `bool` (or a type that
101 can be implicitly converted to `bool`), you can use it in a *predicate
102 assertion* to get the function arguments printed for free:
103 
104 <!-- mdformat off(github rendering does not support multiline tables) -->
105 
106 | Fatal assertion | Nonfatal assertion | Verifies |
107 | --------------------------------- | --------------------------------- | --------------------------- |
108 | `ASSERT_PRED1(pred1, val1)` | `EXPECT_PRED1(pred1, val1)` | `pred1(val1)` is true |
109 | `ASSERT_PRED2(pred2, val1, val2)` | `EXPECT_PRED2(pred2, val1, val2)` | `pred2(val1, val2)` is true |
110 | `...` | `...` | `...` |
111 
112 <!-- mdformat on-->
113 In the above, `predn` is an `n`-ary predicate function or functor, where `val1`,
114 `val2`, ..., and `valn` are its arguments. The assertion succeeds if the
115 predicate returns `true` when applied to the given arguments, and fails
116 otherwise. When the assertion fails, it prints the value of each argument. In
117 either case, the arguments are evaluated exactly once.
118 
119 Here's an example. Given
120 
121 ```c++
122 // Returns true if m and n have no common divisors except 1.
123 bool MutuallyPrime(int m, int n) { ... }
124 
125 const int a = 3;
126 const int b = 4;
127 const int c = 10;
128 ```
129 
130 the assertion
131 
132 ```c++
133  EXPECT_PRED2(MutuallyPrime, a, b);
134 ```
135 
136 will succeed, while the assertion
137 
138 ```c++
139  EXPECT_PRED2(MutuallyPrime, b, c);
140 ```
141 
142 will fail with the message
143 
144 ```none
145 MutuallyPrime(b, c) is false, where
146 b is 4
147 c is 10
148 ```
149 
150 > NOTE:
151 >
152 > 1. If you see a compiler error "no matching function to call" when using
153 > `ASSERT_PRED*` or `EXPECT_PRED*`, please see
154 > [this](faq.md#the-compiler-complains-no-matching-function-to-call-when-i-use-assert-pred-how-do-i-fix-it)
155 > for how to resolve it.

157 #### Using a Function That Returns an AssertionResult
158 
159 While `EXPECT_PRED*()` and friends are handy for a quick job, the syntax is not
160 satisfactory: you have to use different macros for different arities, and it
161 feels more like Lisp than C++. The `::testing::AssertionResult` class solves
162 this problem.
163 
164 An `AssertionResult` object represents the result of an assertion (whether it's
165 a success or a failure, and an associated message). You can create an
166 `AssertionResult` using one of these factory functions:
167 
168 ```c++
169 namespace testing {
170 
171 // Returns an AssertionResult object to indicate that an assertion has
172 // succeeded.
173 AssertionResult AssertionSuccess();
174 
175 // Returns an AssertionResult object to indicate that an assertion has
176 // failed.
177 AssertionResult AssertionFailure();
178 
179 }
180 ```
181 
182 You can then use the `<<` operator to stream messages to the `AssertionResult`
183 object.
184 
185 To provide more readable messages in Boolean assertions (e.g. `EXPECT_TRUE()`),
186 write a predicate function that returns `AssertionResult` instead of `bool`. For
187 example, if you define `IsEven()` as:
188 
189 ```c++
190 ::testing::AssertionResult IsEven(int n) {
191  if ((n % 2) == 0)
192  return ::testing::AssertionSuccess();
193  else
194  return ::testing::AssertionFailure() << n << " is odd";
195 }
196 ```
197 
198 instead of:
199 
200 ```c++
201 bool IsEven(int n) {
202  return (n % 2) == 0;
203 }
204 ```
205 
206 the failed assertion `EXPECT_TRUE(IsEven(Fib(4)))` will print:
207 
208 ```none
209 Value of: IsEven(Fib(4))
210  Actual: false (3 is odd)
211 Expected: true
212 ```
213 
214 instead of a more opaque
215 
216 ```none
217 Value of: IsEven(Fib(4))
218  Actual: false
219 Expected: true
220 ```
221 
222 If you want informative messages in `EXPECT_FALSE` and `ASSERT_FALSE` as well
223 (one third of Boolean assertions in the Google code base are negative ones), and
224 are fine with making the predicate slower in the success case, you can supply a
225 success message:
226 
227 ```c++
228 ::testing::AssertionResult IsEven(int n) {
229  if ((n % 2) == 0)
230  return ::testing::AssertionSuccess() << n << " is even";
231  else
232  return ::testing::AssertionFailure() << n << " is odd";
233 }
234 ```
235 
236 Then the statement `EXPECT_FALSE(IsEven(Fib(6)))` will print
237 
238 ```none
239  Value of: IsEven(Fib(6))
240  Actual: true (8 is even)
241  Expected: false
242 ```

244 #### Using a Predicate-Formatter
245 
246 If you find the default message generated by `(ASSERT|EXPECT)_PRED*` and
247 `(ASSERT|EXPECT)_(TRUE|FALSE)` unsatisfactory, or some arguments to your
248 predicate do not support streaming to `ostream`, you can instead use the
249 following *predicate-formatter assertions* to *fully* customize how the message
250 is formatted:
251 
252 Fatal assertion | Nonfatal assertion | Verifies
253 ------------------------------------------------ | ------------------------------------------------ | --------
254 `ASSERT_PRED_FORMAT1(pred_format1, val1);` | `EXPECT_PRED_FORMAT1(pred_format1, val1);` | `pred_format1(val1)` is successful
255 `ASSERT_PRED_FORMAT2(pred_format2, val1, val2);` | `EXPECT_PRED_FORMAT2(pred_format2, val1, val2);` | `pred_format2(val1, val2)` is successful
256 `...` | `...` | ...
257 
258 The difference between this and the previous group of macros is that instead of
259 a predicate, `(ASSERT|EXPECT)_PRED_FORMAT*` take a *predicate-formatter*
260 (`pred_formatn`), which is a function or functor with the signature:
261 
262 ```c++
263 ::testing::AssertionResult PredicateFormattern(const char* expr1,
264  const char* expr2,
265  ...
266  const char* exprn,
267  T1 val1,
268  T2 val2,
269  ...
270  Tn valn);
271 ```
272 
273 where `val1`, `val2`, ..., and `valn` are the values of the predicate arguments,
274 and `expr1`, `expr2`, ..., and `exprn` are the corresponding expressions as they
275 appear in the source code. The types `T1`, `T2`, ..., and `Tn` can be either
276 value types or reference types. For example, if an argument has type `Foo`, you
277 can declare it as either `Foo` or `const Foo&`, whichever is appropriate.
278 
279 As an example, let's improve the failure message in `MutuallyPrime()`, which was
280 used with `EXPECT_PRED2()`:
281 
282 ```c++
283 // Returns the smallest prime common divisor of m and n,
284 // or 1 when m and n are mutually prime.
285 int SmallestPrimeCommonDivisor(int m, int n) { ... }
286 
287 // A predicate-formatter for asserting that two integers are mutually prime.
288 ::testing::AssertionResult AssertMutuallyPrime(const char* m_expr,
289  const char* n_expr,
290  int m,
291  int n) {
292  if (MutuallyPrime(m, n)) return ::testing::AssertionSuccess();
293 
294  return ::testing::AssertionFailure() << m_expr << " and " << n_expr
295  << " (" << m << " and " << n << ") are not mutually prime, "
296  << "as they have a common divisor " << SmallestPrimeCommonDivisor(m, n);
297 }
298 ```
299 
300 With this predicate-formatter, we can use
301 
302 ```c++
303  EXPECT_PRED_FORMAT2(AssertMutuallyPrime, b, c);
304 ```
305 
306 to generate the message
307 
308 ```none
309 b and c (4 and 10) are not mutually prime, as they have a common divisor 2.
310 ```
311 
312 As you may have realized, many of the built-in assertions we introduced earlier
313 are special cases of `(EXPECT|ASSERT)_PRED_FORMAT*`. In fact, most of them are
314 indeed defined using `(EXPECT|ASSERT)_PRED_FORMAT*`.

316 ### Floating-Point Comparison
317 
318 Comparing floating-point numbers is tricky. Due to round-off errors, it is very
319 unlikely that two floating-points will match exactly. Therefore, `ASSERT_EQ` 's
320 naive comparison usually doesn't work. And since floating-points can have a wide
321 value range, no single fixed error bound works. It's better to compare by a
322 fixed relative error bound, except for values close to 0 due to the loss of
323 precision there.
324 
325 In general, for floating-point comparison to make sense, the user needs to
326 carefully choose the error bound. If they don't want or care to, comparing in
327 terms of Units in the Last Place (ULPs) is a good default, and googletest
328 provides assertions to do this. Full details about ULPs are quite long; if you
329 want to learn more, see
330 [here](https://randomascii.wordpress.com/2012/02/25/comparing-floating-point-numbers-2012-edition/).
331 
332 #### Floating-Point Macros
333 
334 <!-- mdformat off(github rendering does not support multiline tables) -->
335 
336 | Fatal assertion | Nonfatal assertion | Verifies |
337 | ------------------------------- | ------------------------------- | ---------------------------------------- |
338 | `ASSERT_FLOAT_EQ(val1, val2);` | `EXPECT_FLOAT_EQ(val1, val2);` | the two `float` values are almost equal |
339 | `ASSERT_DOUBLE_EQ(val1, val2);` | `EXPECT_DOUBLE_EQ(val1, val2);` | the two `double` values are almost equal |
340 
341 <!-- mdformat on-->
342 
343 By "almost equal" we mean the values are within 4 ULP's from each other.
344 
345 The following assertions allow you to choose the acceptable error bound:
346 
347 <!-- mdformat off(github rendering does not support multiline tables) -->
348 
349 | Fatal assertion | Nonfatal assertion | Verifies |
350 | ------------------------------------- | ------------------------------------- | -------------------------------------------------------------------------------- |
351 | `ASSERT_NEAR(val1, val2, abs_error);` | `EXPECT_NEAR(val1, val2, abs_error);` | the difference between `val1` and `val2` doesn't exceed the given absolute error |
352

355 #### Floating-Point Predicate-Format Functions
356 
357 Some floating-point operations are useful, but not that often used. In order to
358 avoid an explosion of new macros, we provide them as predicate-format functions
359 that can be used in predicate assertion macros (e.g. `EXPECT_PRED_FORMAT2`,
360 etc).
361 
362 ```c++
363 EXPECT_PRED_FORMAT2(::testing::FloatLE, val1, val2);
364 EXPECT_PRED_FORMAT2(::testing::DoubleLE, val1, val2);
365 ```
366 
367 Verifies that `val1` is less than, or almost equal to, `val2`. You can replace
368 `EXPECT_PRED_FORMAT2` in the above table with `ASSERT_PRED_FORMAT2`.

370 ### Asserting Using gMock Matchers
371 
372 [gMock](../../googlemock) comes with a library of matchers for validating
373 arguments passed to mock objects. A gMock *matcher* is basically a predicate
374 that knows how to describe itself. It can be used in these assertion macros:
375 
376 <!-- mdformat off(github rendering does not support multiline tables) -->
377 
378 | Fatal assertion | Nonfatal assertion | Verifies |
379 | ------------------------------ | ------------------------------ | --------------------- |
380 | `ASSERT_THAT(value, matcher);` | `EXPECT_THAT(value, matcher);` | value matches matcher |
381 
382 <!-- mdformat on-->
383 
384 For example, `StartsWith(prefix)` is a matcher that matches a string starting
385 with `prefix`, and you can write:
386 
387 ```c++
388 using ::testing::StartsWith;
389 ...
390  // Verifies that Foo() returns a string starting with "Hello".
391  EXPECT_THAT(Foo(), StartsWith("Hello"));
392 ```
393 
394 Read this
395 [recipe](../../googlemock/docs/cook_book.md#using-matchers-in-googletest-assertions)
396 in the gMock Cookbook for more details.
397 
398 gMock has a rich set of matchers. You can do many things googletest cannot do
399 alone with them. For a list of matchers gMock provides, read
400 [this](../../googlemock/docs/cook_book.md##using-matchers). It's easy to write
401 your [own matchers](../../googlemock/docs/cook_book.md#NewMatchers) too.
402 
403 gMock is bundled with googletest, so you don't need to add any build dependency
404 in order to take advantage of this. Just include `"gmock/gmock.h"`
405 and you're ready to go.

411

413 [string matchers](../../googlemock/docs/cheat_sheet.md#string-matchers) with

416 
417 ```c++

419 using ::testing::MatchesRegex;
420 ...
421  ASSERT_THAT(foo_string, HasSubstr("needle"));
422  EXPECT_THAT(bar_string, MatchesRegex("\\w*\\d+"));

431 #include "template/prototemplate/testing/xpath_matcher.h"
432 using prototemplate::testing::MatchesXPath;
433 EXPECT_THAT(html_string, MatchesXPath("//a[text()='click here']"));

437 
438 These assertions test for `HRESULT` success or failure.
439 
440 Fatal assertion | Nonfatal assertion | Verifies
441 -------------------------------------- | -------------------------------------- | --------
442 `ASSERT_HRESULT_SUCCEEDED(expression)` | `EXPECT_HRESULT_SUCCEEDED(expression)` | `expression` is a success `HRESULT`

457 ### Type Assertions
458 
459 You can call the function
460 
461 ```c++
462 ::testing::StaticAssertTypeEq<T1, T2>();
463 ```
464 
465 to assert that types `T1` and `T2` are the same. The function does nothing if
466 the assertion is satisfied. If the types are different, the function call will
467 fail to compile, the compiler error message will say that
468 `T1 and T2 are not the same type` and most likely (depending on the compiler)
469 show you the actual values of `T1` and `T2`. This is mainly useful inside
470 template code.
471 
472 **Caveat**: When used inside a member function of a class template or a function
473 template, `StaticAssertTypeEq<T1, T2>()` is effective only if the function is
474 instantiated. For example, given:
475 
476 ```c++
477 template <typename T> class Foo {
478  public:
479  void Bar() { ::testing::StaticAssertTypeEq<int, T>(); }
480 };
481 ```
482 
483 the code:
484 
485 ```c++
486 void Test1() { Foo<bool> foo; }
487 ```
488 
489 will not generate a compiler error, as `Foo<bool>::Bar()` is never actually
490 instantiated. Instead, you need:
491 
492 ```c++
493 void Test2() { Foo<bool> foo; foo.Bar(); }
494 ```
495 
496 to cause a compiler error.

498 ### Assertion Placement
499 
500 You can use assertions in any C++ function. In particular, it doesn't have to be
501 a method of the test fixture class. The one constraint is that assertions that
502 generate a fatal failure (`FAIL*` and `ASSERT_*`) can only be used in
503 void-returning functions. This is a consequence of Google's not using
504 exceptions. By placing it in a non-void function you'll get a confusing compile
505 error like `"error: void value not ignored as it ought to be"` or `"cannot
506 initialize return object of type 'bool' with an rvalue of type 'void'"` or
507 `"error: no viable conversion from 'void' to 'string'"`.
508 
509 If you need to use fatal assertions in a function that returns non-void, one
510 option is to make the function return the value in an out parameter instead. For
511 example, you can rewrite `T2 Foo(T1 x)` to `void Foo(T1 x, T2* result)`. You
512 need to make sure that `*result` contains some sensible value even when the
513 function returns prematurely. As the function now returns `void`, you can use
514 any assertion inside of it.
515 
516 If changing the function's type is not an option, you should just use assertions
517 that generate non-fatal failures, such as `ADD_FAILURE*` and `EXPECT_*`.
518 
519 NOTE: Constructors and destructors are not considered void-returning functions,
520 according to the C++ language specification, and so you may not use fatal
521 assertions in them; you'll get a compilation error if you try. Instead, either
522 call `abort` and crash the entire test executable, or put the fatal assertion in
523 a `SetUp`/`TearDown` function; see
524 [constructor/destructor vs. `SetUp`/`TearDown`](faq.md#CtorVsSetUp)
525 
526 WARNING: A fatal assertion in a helper function (private void-returning method)
527 called from a constructor or destructor does not does not terminate the current
528 test, as your intuition might suggest: it merely returns from the constructor or
529 destructor early, possibly leaving your object in a partially-constructed or
530 partially-destructed state! You almost certainly want to `abort` or use
531 `SetUp`/`TearDown` instead.

533 ## Teaching googletest How to Print Your Values
534 
535 When a test assertion such as `EXPECT_EQ` fails, googletest prints the argument
536 values to help you debug. It does this using a user-extensible value printer.
537 
538 This printer knows how to print built-in C++ types, native arrays, STL
539 containers, and any type that supports the `<<` operator. For other types, it
540 prints the raw bytes in the value and hopes that you the user can figure it out.
541 
542 As mentioned earlier, the printer is *extensible*. That means you can teach it
543 to do a better job at printing your particular type than to dump the bytes. To
544 do that, define `<<` for your type:
545 
546 ```c++
547 #include <ostream>
548 
549 namespace foo {
550 
551 class Bar { // We want googletest to be able to print instances of this.
552 ...
553  // Create a free inline friend function.
554  friend std::ostream& operator<<(std::ostream& os, const Bar& bar) {
555  return os << bar.DebugString(); // whatever needed to print bar to os
556  }
557 };
558 
559 // If you can't declare the function in the class it's important that the
560 // << operator is defined in the SAME namespace that defines Bar. C++'s look-up
561 // rules rely on that.
562 std::ostream& operator<<(std::ostream& os, const Bar& bar) {
563  return os << bar.DebugString(); // whatever needed to print bar to os
564 }
565 
566 } // namespace foo
567 ```
568 
569 Sometimes, this might not be an option: your team may consider it bad style to
570 have a `<<` operator for `Bar`, or `Bar` may already have a `<<` operator that
571 doesn't do what you want (and you cannot change it). If so, you can instead
572 define a `PrintTo()` function like this:
573 
574 ```c++
575 #include <ostream>
576 
577 namespace foo {
578 
579 class Bar {
580  ...
581  friend void PrintTo(const Bar& bar, std::ostream* os) {
582  *os << bar.DebugString(); // whatever needed to print bar to os
583  }
584 };
585 
586 // If you can't declare the function in the class it's important that PrintTo()
587 // is defined in the SAME namespace that defines Bar. C++'s look-up rules rely
588 // on that.
589 void PrintTo(const Bar& bar, std::ostream* os) {
590  *os << bar.DebugString(); // whatever needed to print bar to os
591 }
592 
593 } // namespace foo
594 ```
595 
596 If you have defined both `<<` and `PrintTo()`, the latter will be used when
597 googletest is concerned. This allows you to customize how the value appears in
598 googletest's output without affecting code that relies on the behavior of its
599 `<<` operator.
600 
601 If you want to print a value `x` using googletest's value printer yourself, just
602 call `::testing::PrintToString(x)`, which returns an `std::string`:
603 
604 ```c++
605 vector<pair<Bar, int> > bar_ints = GetBarIntVector();
606 
607 EXPECT_TRUE(IsCorrectBarIntVector(bar_ints))
608  << "bar_ints = " << ::testing::PrintToString(bar_ints);
609 ```

611 ## Death Tests
612 
613 In many applications, there are assertions that can cause application failure if
614 a condition is not met. These sanity checks, which ensure that the program is in
615 a known good state, are there to fail at the earliest possible time after some
616 program state is corrupted. If the assertion checks the wrong condition, then
617 the program may proceed in an erroneous state, which could lead to memory
618 corruption, security holes, or worse. Hence it is vitally important to test that
619 such assertion statements work as expected.
620 
621 Since these precondition checks cause the processes to die, we call such tests
622 _death tests_. More generally, any test that checks that a program terminates
623 (except by throwing an exception) in an expected fashion is also a death test.
624 
625 Note that if a piece of code throws an exception, we don't consider it "death"
626 for the purpose of death tests, as the caller of the code could catch the
627 exception and avoid the crash. If you want to verify exceptions thrown by your
628 code, see [Exception Assertions](#ExceptionAssertions).
629 
630 If you want to test `EXPECT_*()/ASSERT_*()` failures in your test code, see
631 Catching Failures
632 
633 ### How to Write a Death Test
634 
635 googletest has the following macros to support death tests:
636 
637 Fatal assertion | Nonfatal assertion | Verifies
638 ------------------------------------------------ | ------------------------------------------------ | --------
639 `ASSERT_DEATH(statement, matcher);` | `EXPECT_DEATH(statement, matcher);` | `statement` crashes with the given error
640 `ASSERT_DEATH_IF_SUPPORTED(statement, matcher);` | `EXPECT_DEATH_IF_SUPPORTED(statement, matcher);` | if death tests are supported, verifies that `statement` crashes with the given error; otherwise verifies nothing
641 `ASSERT_DEBUG_DEATH(statement, matcher);` | `EXPECT_DEBUG_DEATH(statement, matcher);` | `statement` crashes with the given error **in debug mode**. When not in debug (i.e. `NDEBUG` is defined), this just executes `statement`
642 `ASSERT_EXIT(statement, predicate, matcher);` | `EXPECT_EXIT(statement, predicate, matcher);` | `statement` exits with the given error, and its exit code matches `predicate`
643 
644 where `statement` is a statement that is expected to cause the process to die,
645 `predicate` is a function or function object that evaluates an integer exit
646 status, and `matcher` is either a gMock matcher matching a `const std::string&`
647 or a (Perl) regular expression - either of which is matched against the stderr
648 output of `statement`. For legacy reasons, a bare string (i.e. with no matcher)
649 is interpreted as `ContainsRegex(str)`, **not** `Eq(str)`. Note that `statement`
650 can be *any valid statement* (including *compound statement*) and doesn't have
651 to be an expression.
652 
653 As usual, the `ASSERT` variants abort the current test function, while the
654 `EXPECT` variants do not.
655 
656 > NOTE: We use the word "crash" here to mean that the process terminates with a
657 > *non-zero* exit status code. There are two possibilities: either the process
658 > has called `exit()` or `_exit()` with a non-zero value, or it may be killed by
659 > a signal.
660 >
661 > This means that if *`statement`* terminates the process with a 0 exit code, it
662 > is *not* considered a crash by `EXPECT_DEATH`. Use `EXPECT_EXIT` instead if
663 > this is the case, or if you want to restrict the exit code more precisely.
664 
665 A predicate here must accept an `int` and return a `bool`. The death test
666 succeeds only if the predicate returns `true`. googletest defines a few
667 predicates that handle the most common cases:
668 
669 ```c++
670 ::testing::ExitedWithCode(exit_code)
671 ```
672 
673 This expression is `true` if the program exited normally with the given exit
674 code.
675 
676 ```c++
677 ::testing::KilledBySignal(signal_number) // Not available on Windows.
678 ```
679 
680 This expression is `true` if the program was killed by the given signal.
681 
682 The `*_DEATH` macros are convenient wrappers for `*_EXIT` that use a predicate
683 that verifies the process' exit code is non-zero.
684 
685 Note that a death test only cares about three things:
686 
687 1. does `statement` abort or exit the process?
688 2. (in the case of `ASSERT_EXIT` and `EXPECT_EXIT`) does the exit status
689  satisfy `predicate`? Or (in the case of `ASSERT_DEATH` and `EXPECT_DEATH`)
690  is the exit status non-zero? And
691 3. does the stderr output match `matcher`?
692 
693 In particular, if `statement` generates an `ASSERT_*` or `EXPECT_*` failure, it
694 will **not** cause the death test to fail, as googletest assertions don't abort
695 the process.
696 
697 To write a death test, simply use one of the above macros inside your test
698 function. For example,
699 
700 ```c++
701 TEST(MyDeathTest, Foo) {
702  // This death test uses a compound statement.
703  ASSERT_DEATH({
704  int n = 5;
705  Foo(&n);
706  }, "Error on line .* of Foo()");
707 }
708 
709 TEST(MyDeathTest, NormalExit) {
710  EXPECT_EXIT(NormalExit(), ::testing::ExitedWithCode(0), "Success");
711 }
712 
713 TEST(MyDeathTest, KillMyself) {
714  EXPECT_EXIT(KillMyself(), ::testing::KilledBySignal(SIGKILL),
715  "Sending myself unblockable signal");
716 }
717 ```
718 
719 verifies that:
720 
721 * calling `Foo(5)` causes the process to die with the given error message,
722 * calling `NormalExit()` causes the process to print `"Success"` to stderr and
723  exit with exit code 0, and
724 * calling `KillMyself()` kills the process with signal `SIGKILL`.
725 
726 The test function body may contain other assertions and statements as well, if
727 necessary.

754 ### Regular Expression Syntax
755 
756 On POSIX systems (e.g. Linux, Cygwin, and Mac), googletest uses the
757 [POSIX extended regular expression](http://www.opengroup.org/onlinepubs/009695399/basedefs/xbd_chap09.html#tag_09_04)
758 syntax. To learn about this syntax, you may want to read this
759 [Wikipedia entry](http://en.wikipedia.org/wiki/Regular_expression#POSIX_Extended_Regular_Expressions).
760 
761 On Windows, googletest uses its own simple regular expression implementation. It
762 lacks many features. For example, we don't support union (`"x|y"`), grouping
763 (`"(xy)"`), brackets (`"[xy]"`), and repetition count (`"x{5,7}"`), among
764 others. Below is what we do support (`A` denotes a literal character, period
765 (`.`), or a single `\\ ` escape sequence; `x` and `y` denote regular
766 expressions.):
767 
768 Expression | Meaning
769 ---------- | --------------------------------------------------------------
770 `c` | matches any literal character `c`
771 `\\d` | matches any decimal digit
772 `\\D` | matches any character that's not a decimal digit
773 `\\f` | matches `\f`
774 `\\n` | matches `\n`
775 `\\r` | matches `\r`
776 `\\s` | matches any ASCII whitespace, including `\n`
777 `\\S` | matches any character that's not a whitespace
778 `\\t` | matches `\t`
779 `\\v` | matches `\v`
780 `\\w` | matches any letter, `_`, or decimal digit
781 `\\W` | matches any character that `\\w` doesn't match
782 `\\c` | matches any literal character `c`, which must be a punctuation
783 `.` | matches any single character except `\n`
784 `A?` | matches 0 or 1 occurrences of `A`
785 `A*` | matches 0 or many occurrences of `A`
786 `A+` | matches 1 or many occurrences of `A`
787 `^` | matches the beginning of a string (not that of each line)
788 `$` | matches the end of a string (not that of each line)
789 `xy` | matches `x` followed by `y`
790 
791 To help you determine which capability is available on your system, googletest
792 defines macros to govern which regular expression it is using. The macros are:
793 `GTEST_USES_SIMPLE_RE=1` or `GTEST_USES_POSIX_RE=1`. If you want your death
794 tests to work in all cases, you can either `#if` on these macros or use the more
795 limited syntax only.

797 ### How It Works
798 
799 Under the hood, `ASSERT_EXIT()` spawns a new process and executes the death test
800 statement in that process. The details of how precisely that happens depend on
801 the platform and the variable ::testing::GTEST_FLAG(death_test_style) (which is
802 initialized from the command-line flag `--gtest_death_test_style`).
803 
804 * On POSIX systems, `fork()` (or `clone()` on Linux) is used to spawn the
805  child, after which:
806  * If the variable's value is `"fast"`, the death test statement is
807  immediately executed.
808  * If the variable's value is `"threadsafe"`, the child process re-executes
809  the unit test binary just as it was originally invoked, but with some
810  extra flags to cause just the single death test under consideration to
811  be run.
812 * On Windows, the child is spawned using the `CreateProcess()` API, and
813  re-executes the binary to cause just the single death test under
814  consideration to be run - much like the `threadsafe` mode on POSIX.
815 
816 Other values for the variable are illegal and will cause the death test to fail.
817 Currently, the flag's default value is **"fast"**
818 
819 1. the child's exit status satisfies the predicate, and
820 2. the child's stderr matches the regular expression.
821 
822 If the death test statement runs to completion without dying, the child process
823 will nonetheless terminate, and the assertion fails.
824 
825 ### Death Tests And Threads
826 
827 The reason for the two death test styles has to do with thread safety. Due to
828 well-known problems with forking in the presence of threads, death tests should
829 be run in a single-threaded context. Sometimes, however, it isn't feasible to
830 arrange that kind of environment. For example, statically-initialized modules
831 may start threads before main is ever reached. Once threads have been created,
832 it may be difficult or impossible to clean them up.
833 
834 googletest has three features intended to raise awareness of threading issues.
835 
836 1. A warning is emitted if multiple threads are running when a death test is
837  encountered.
838 2. Test suites with a name ending in "DeathTest" are run before all other
839  tests.
840 3. It uses `clone()` instead of `fork()` to spawn the child process on Linux
841  (`clone()` is not available on Cygwin and Mac), as `fork()` is more likely
842  to cause the child to hang when the parent process has multiple threads.
843 
844 It's perfectly fine to create threads inside a death test statement; they are
845 executed in a separate process and cannot affect the parent.
846 
847 ### Death Test Styles
848 
849 The "threadsafe" death test style was introduced in order to help mitigate the
850 risks of testing in a possibly multithreaded environment. It trades increased
851 test execution time (potentially dramatically so) for improved thread safety.
852 
853 The automated testing framework does not set the style flag. You can choose a
854 particular style of death tests by setting the flag programmatically:
855 
856 ```c++
857 testing::FLAGS_gtest_death_test_style="threadsafe"
858 ```
859 
860 You can do this in `main()` to set the style for all death tests in the binary,
861 or in individual tests. Recall that flags are saved before running each test and
862 restored afterwards, so you need not do that yourself. For example:
863 
864 ```c++
865 int main(int argc, char** argv) {
866  ::testing::InitGoogleTest(&argc, argv);
867  ::testing::FLAGS_gtest_death_test_style = "fast";
868  return RUN_ALL_TESTS();
869 }
870 
871 TEST(MyDeathTest, TestOne) {
872  ::testing::FLAGS_gtest_death_test_style = "threadsafe";
873  // This test is run in the "threadsafe" style:
874  ASSERT_DEATH(ThisShouldDie(), "");
875 }
876 
877 TEST(MyDeathTest, TestTwo) {
878  // This test is run in the "fast" style:
879  ASSERT_DEATH(ThisShouldDie(), "");
880 }
881 ```
882 
883 ### Caveats
884 
885 The `statement` argument of `ASSERT_EXIT()` can be any valid C++ statement. If
886 it leaves the current function via a `return` statement or by throwing an
887 exception, the death test is considered to have failed. Some googletest macros
888 may return from the current function (e.g. `ASSERT_TRUE()`), so be sure to avoid
889 them in `statement`.
890 
891 Since `statement` runs in the child process, any in-memory side effect (e.g.
892 modifying a variable, releasing memory, etc) it causes will *not* be observable
893 in the parent process. In particular, if you release memory in a death test,
894 your program will fail the heap check as the parent process will never see the
895 memory reclaimed. To solve this problem, you can
896 
897 1. try not to free memory in a death test;
898 2. free the memory again in the parent process; or
899 3. do not use the heap checker in your program.
900 
901 Due to an implementation detail, you cannot place multiple death test assertions
902 on the same line; otherwise, compilation will fail with an unobvious error
903 message.
904 
905 Despite the improved thread safety afforded by the "threadsafe" style of death
906 test, thread problems such as deadlock are still possible in the presence of
907 handlers registered with `pthread_atfork(3)`.

912 ### Adding Traces to Assertions
913 
914 If a test sub-routine is called from several places, when an assertion inside it
915 fails, it can be hard to tell which invocation of the sub-routine the failure is
916 from. You can alleviate this problem using extra logging or custom failure
917 messages, but that usually clutters up your tests. A better solution is to use
918 the `SCOPED_TRACE` macro or the `ScopedTrace` utility:
919 
920 ```c++
921 SCOPED_TRACE(message);
922 ```
923 ```c++
924 ScopedTrace trace("file_path", line_number, message);
925 ```
926 
927 where `message` can be anything streamable to `std::ostream`. `SCOPED_TRACE`
928 macro will cause the current file name, line number, and the given message to be
929 added in every failure message. `ScopedTrace` accepts explicit file name and
930 line number in arguments, which is useful for writing test helpers. The effect
931 will be undone when the control leaves the current lexical scope.
932 
933 For example,
934 
935 ```c++
936 10: void Sub1(int n) {
937 11: EXPECT_EQ(Bar(n), 1);
938 12: EXPECT_EQ(Bar(n + 1), 2);
939 13: }
940 14:
941 15: TEST(FooTest, Bar) {
942 16: {
943 17: SCOPED_TRACE("A"); // This trace point will be included in
944 18: // every failure in this scope.
945 19: Sub1(1);
946 20: }
947 21: // Now it won't.
948 22: Sub1(9);
949 23: }
950 ```
951 
952 could result in messages like these:
953 
954 ```none
955 path/to/foo_test.cc:11: Failure
956 Value of: Bar(n)
957 Expected: 1
958  Actual: 2
959 Google Test trace:
960 path/to/foo_test.cc:17: A
961 
962 path/to/foo_test.cc:12: Failure
963 Value of: Bar(n + 1)
964 Expected: 2
965  Actual: 3
966 ```
967 
968 Without the trace, it would've been difficult to know which invocation of
969 `Sub1()` the two failures come from respectively. (You could add an extra
970 message to each assertion in `Sub1()` to indicate the value of `n`, but that's
971 tedious.)
972 
973 Some tips on using `SCOPED_TRACE`:
974 
975 1. With a suitable message, it's often enough to use `SCOPED_TRACE` at the
976  beginning of a sub-routine, instead of at each call site.
977 2. When calling sub-routines inside a loop, make the loop iterator part of the
978  message in `SCOPED_TRACE` such that you can know which iteration the failure
979  is from.
980 3. Sometimes the line number of the trace point is enough for identifying the
981  particular invocation of a sub-routine. In this case, you don't have to
982  choose a unique message for `SCOPED_TRACE`. You can simply use `""`.
983 4. You can use `SCOPED_TRACE` in an inner scope when there is one in the outer
984  scope. In this case, all active trace points will be included in the failure
985  messages, in reverse order they are encountered.
986 5. The trace dump is clickable in Emacs - hit `return` on a line number and
987  you'll be taken to that line in the source file!

989 ### Propagating Fatal Failures
990 
991 A common pitfall when using `ASSERT_*` and `FAIL*` is not understanding that
992 when they fail they only abort the _current function_, not the entire test. For
993 example, the following test will segfault:
994 
995 ```c++
996 void Subroutine() {
997  // Generates a fatal failure and aborts the current function.
998  ASSERT_EQ(1, 2);
999 
1000  // The following won't be executed.
1001  ...
1002 }
1003 
1004 TEST(FooTest, Bar) {
1005  Subroutine(); // The intended behavior is for the fatal failure
1006  // in Subroutine() to abort the entire test.
1007 
1008  // The actual behavior: the function goes on after Subroutine() returns.
1009  int* p = nullptr;
1010  *p = 3; // Segfault!
1011 }
1012 ```
1013 
1014 To alleviate this, googletest provides three different solutions. You could use
1015 either exceptions, the `(ASSERT|EXPECT)_NO_FATAL_FAILURE` assertions or the
1016 `HasFatalFailure()` function. They are described in the following two
1017 subsections.
1018 
1019 #### Asserting on Subroutines with an exception
1020 
1021 The following code can turn ASSERT-failure into an exception:
1022 
1023 ```c++
1024 class ThrowListener : public testing::EmptyTestEventListener {
1025  void OnTestPartResult(const testing::TestPartResult& result) override {
1026  if (result.type() == testing::TestPartResult::kFatalFailure) {
1027  throw testing::AssertionException(result);
1028  }
1029  }
1030 };
1031 int main(int argc, char** argv) {
1032  ...
1033  testing::UnitTest::GetInstance()->listeners().Append(new ThrowListener);
1034  return RUN_ALL_TESTS();
1035 }
1036 ```
1037 
1038 This listener should be added after other listeners if you have any, otherwise
1039 they won't see failed `OnTestPartResult`.
1040 
1041 #### Asserting on Subroutines
1042 
1043 As shown above, if your test calls a subroutine that has an `ASSERT_*` failure
1044 in it, the test will continue after the subroutine returns. This may not be what
1045 you want.
1046 
1047 Often people want fatal failures to propagate like exceptions. For that
1048 googletest offers the following macros:
1049 
1050 Fatal assertion | Nonfatal assertion | Verifies
1051 ------------------------------------- | ------------------------------------- | --------
1052 `ASSERT_NO_FATAL_FAILURE(statement);` | `EXPECT_NO_FATAL_FAILURE(statement);` | `statement` doesn't generate any new fatal failures in the current thread.
1053 
1054 Only failures in the thread that executes the assertion are checked to determine
1055 the result of this type of assertions. If `statement` creates new threads,
1056 failures in these threads are ignored.
1057 
1058 Examples:
1059 
1060 ```c++
1061 ASSERT_NO_FATAL_FAILURE(Foo());
1062 
1063 int i;
1064 EXPECT_NO_FATAL_FAILURE({
1065  i = Bar();
1066 });
1067 ```
1068 
1069 Assertions from multiple threads are currently not supported on Windows.
1070 
1071 #### Checking for Failures in the Current Test
1072 
1073 `HasFatalFailure()` in the `::testing::Test` class returns `true` if an
1074 assertion in the current test has suffered a fatal failure. This allows
1075 functions to catch fatal failures in a sub-routine and return early.
1076 
1077 ```c++
1078 class Test {
1079  public:
1080  ...
1081  static bool HasFatalFailure();
1082 };
1083 ```
1084 
1085 The typical usage, which basically simulates the behavior of a thrown exception,
1086 is:
1087 
1088 ```c++
1089 TEST(FooTest, Bar) {
1090  Subroutine();
1091  // Aborts if Subroutine() had a fatal failure.
1092  if (HasFatalFailure()) return;
1093 
1094  // The following won't be executed.
1095  ...
1096 }
1097 ```
1098 
1099 If `HasFatalFailure()` is used outside of `TEST()` , `TEST_F()` , or a test
1100 fixture, you must add the `::testing::Test::` prefix, as in:
1101 
1102 ```c++
1103 if (::testing::Test::HasFatalFailure()) return;
1104 ```
1105 
1106 Similarly, `HasNonfatalFailure()` returns `true` if the current test has at
1107 least one non-fatal failure, and `HasFailure()` returns `true` if the current
1108 test has at least one failure of either kind.

1110 ## Logging Additional Information
1111 
1112 In your test code, you can call `RecordProperty("key", value)` to log additional
1113 information, where `value` can be either a string or an `int`. The *last* value
1114 recorded for a key will be emitted to the
1115 [XML output](#generating-an-xml-report) if you specify one. For example, the
1116 test
1117 
1118 ```c++
1119 TEST_F(WidgetUsageTest, MinAndMaxWidgets) {
1120  RecordProperty("MaximumWidgets", ComputeMaxUsage());
1121  RecordProperty("MinimumWidgets", ComputeMinUsage());
1122 }
1123 ```
1124 
1125 will output XML like this:
1126 
1127 ```xml
1128  ...
1129  <testcase name="MinAndMaxWidgets" status="run" time="0.006" classname="WidgetUsageTest" MaximumWidgets="12" MinimumWidgets="9" />
1130  ...
1131 ```
1132 
1133 > NOTE:
1134 >
1135 > * `RecordProperty()` is a static member of the `Test` class. Therefore it
1136 > needs to be prefixed with `::testing::Test::` if used outside of the
1137 > `TEST` body and the test fixture class.
1138 > * *`key`* must be a valid XML attribute name, and cannot conflict with the
1139 > ones already used by googletest (`name`, `status`, `time`, `classname`,
1140 > `type_param`, and `value_param`).
1141 > * Calling `RecordProperty()` outside of the lifespan of a test is allowed.
1142 > If it's called outside of a test but between a test suite's
1143 > `SetUpTestSuite()` and `TearDownTestSuite()` methods, it will be
1144 > attributed to the XML element for the test suite. If it's called outside
1145 > of all test suites (e.g. in a test environment), it will be attributed to
1146 > the top-level XML element.

1148 ## Sharing Resources Between Tests in the Same Test Suite
1149 
1150 googletest creates a new test fixture object for each test in order to make
1151 tests independent and easier to debug. However, sometimes tests use resources
1152 that are expensive to set up, making the one-copy-per-test model prohibitively
1153 expensive.
1154 
1155 If the tests don't change the resource, there's no harm in their sharing a
1156 single resource copy. So, in addition to per-test set-up/tear-down, googletest
1157 also supports per-test-suite set-up/tear-down. To use it:
1158 
1159 1. In your test fixture class (say `FooTest` ), declare as `static` some member
1160  variables to hold the shared resources.
1161 2. Outside your test fixture class (typically just below it), define those
1162  member variables, optionally giving them initial values.
1163 3. In the same test fixture class, define a `static void SetUpTestSuite()`
1164  function (remember not to spell it as **`SetupTestSuite`** with a small
1165  `u`!) to set up the shared resources and a `static void TearDownTestSuite()`
1166  function to tear them down.
1167 
1168 That's it! googletest automatically calls `SetUpTestSuite()` before running the
1169 *first test* in the `FooTest` test suite (i.e. before creating the first
1170 `FooTest` object), and calls `TearDownTestSuite()` after running the *last test*
1171 in it (i.e. after deleting the last `FooTest` object). In between, the tests can
1172 use the shared resources.
1173 
1174 Remember that the test order is undefined, so your code can't depend on a test
1175 preceding or following another. Also, the tests must either not modify the state
1176 of any shared resource, or, if they do modify the state, they must restore the
1177 state to its original value before passing control to the next test.
1178 
1179 Here's an example of per-test-suite set-up and tear-down:
1180 
1181 ```c++
1182 class FooTest : public ::testing::Test {
1183  protected:
1184  // Per-test-suite set-up.
1185  // Called before the first test in this test suite.
1186  // Can be omitted if not needed.
1187  static void SetUpTestSuite() {
1188  shared_resource_ = new ...;
1189  }
1190 
1191  // Per-test-suite tear-down.
1192  // Called after the last test in this test suite.
1193  // Can be omitted if not needed.
1194  static void TearDownTestSuite() {
1195  delete shared_resource_;
1196  shared_resource_ = nullptr;
1197  }
1198 
1199  // You can define per-test set-up logic as usual.
1200  virtual void SetUp() { ... }
1201 
1202  // You can define per-test tear-down logic as usual.
1203  virtual void TearDown() { ... }
1204 
1205  // Some expensive resource shared by all tests.
1206  static T* shared_resource_;
1207 };
1208 
1209 T* FooTest::shared_resource_ = nullptr;
1210 
1211 TEST_F(FooTest, Test1) {
1212  ... you can refer to shared_resource_ here ...
1213 }
1214 
1215 TEST_F(FooTest, Test2) {
1216  ... you can refer to shared_resource_ here ...
1217 }
1218 ```
1219 
1220 NOTE: Though the above code declares `SetUpTestSuite()` protected, it may
1221 sometimes be necessary to declare it public, such as when using it with
1222 `TEST_P`.

1224 ## Global Set-Up and Tear-Down
1225 
1226 Just as you can do set-up and tear-down at the test level and the test suite
1227 level, you can also do it at the test program level. Here's how.
1228 
1229 First, you subclass the `::testing::Environment` class to define a test
1230 environment, which knows how to set-up and tear-down:
1231 
1232 ```c++
1233 class Environment : public ::testing::Environment {
1234  public:
1235  ~Environment() override {}
1236 
1237  // Override this to define how to set up the environment.
1238  void SetUp() override {}
1239 
1240  // Override this to define how to tear down the environment.
1241  void TearDown() override {}
1242 };
1243 ```
1244 
1245 Then, you register an instance of your environment class with googletest by
1246 calling the `::testing::AddGlobalTestEnvironment()` function:
1247 
1248 ```c++
1249 Environment* AddGlobalTestEnvironment(Environment* env);
1250 ```
1251 
1252 Now, when `RUN_ALL_TESTS()` is called, it first calls the `SetUp()` method of
1253 each environment object, then runs the tests if none of the environments
1254 reported fatal failures and `GTEST_SKIP()` was not called. `RUN_ALL_TESTS()`
1255 always calls `TearDown()` with each environment object, regardless of whether or
1256 not the tests were run.
1257 
1258 It's OK to register multiple environment objects. In this suite, their `SetUp()`
1259 will be called in the order they are registered, and their `TearDown()` will be
1260 called in the reverse order.
1261 
1262 Note that googletest takes ownership of the registered environment objects.
1263 Therefore **do not delete them** by yourself.
1264 
1265 You should call `AddGlobalTestEnvironment()` before `RUN_ALL_TESTS()` is called,
1266 probably in `main()`. If you use `gtest_main`, you need to call this before
1267 `main()` starts for it to take effect. One way to do this is to define a global
1268 variable like this:
1269 
1270 ```c++
1271 ::testing::Environment* const foo_env =
1272  ::testing::AddGlobalTestEnvironment(new FooEnvironment);
1273 ```
1274 
1275 However, we strongly recommend you to write your own `main()` and call
1276 `AddGlobalTestEnvironment()` there, as relying on initialization of global
1277 variables makes the code harder to read and may cause problems when you register
1278 multiple environments from different translation units and the environments have
1279 dependencies among them (remember that the compiler doesn't guarantee the order
1280 in which global variables from different translation units are initialized).

1282 ## Value-Parameterized Tests
1283 
1284 *Value-parameterized tests* allow you to test your code with different
1285 parameters without writing multiple copies of the same test. This is useful in a
1286 number of situations, for example:
1287 
1288 * You have a piece of code whose behavior is affected by one or more
1289  command-line flags. You want to make sure your code performs correctly for
1290  various values of those flags.
1291 * You want to test different implementations of an OO interface.
1292 * You want to test your code over various inputs (a.k.a. data-driven testing).
1293  This feature is easy to abuse, so please exercise your good sense when doing
1294  it!
1295 
1296 ### How to Write Value-Parameterized Tests
1297 
1298 To write value-parameterized tests, first you should define a fixture class. It
1299 must be derived from both `testing::Test` and `testing::WithParamInterface<T>`
1300 (the latter is a pure interface), where `T` is the type of your parameter
1301 values. For convenience, you can just derive the fixture class from
1302 `testing::TestWithParam<T>`, which itself is derived from both `testing::Test`
1303 and `testing::WithParamInterface<T>`. `T` can be any copyable type. If it's a
1304 raw pointer, you are responsible for managing the lifespan of the pointed
1305 values.
1306 
1307 NOTE: If your test fixture defines `SetUpTestSuite()` or `TearDownTestSuite()`
1308 they must be declared **public** rather than **protected** in order to use
1309 `TEST_P`.
1310 
1311 ```c++
1312 class FooTest :
1313  public testing::TestWithParam<const char*> {
1314  // You can implement all the usual fixture class members here.
1315  // To access the test parameter, call GetParam() from class
1316  // TestWithParam<T>.
1317 };
1318 
1319 // Or, when you want to add parameters to a pre-existing fixture class:
1320 class BaseTest : public testing::Test {
1321  ...
1322 };
1323 class BarTest : public BaseTest,
1324  public testing::WithParamInterface<const char*> {
1325  ...
1326 };
1327 ```
1328 
1329 Then, use the `TEST_P` macro to define as many test patterns using this fixture
1330 as you want. The `_P` suffix is for "parameterized" or "pattern", whichever you
1331 prefer to think.
1332 
1333 ```c++
1334 TEST_P(FooTest, DoesBlah) {
1335  // Inside a test, access the test parameter with the GetParam() method
1336  // of the TestWithParam<T> class:
1337  EXPECT_TRUE(foo.Blah(GetParam()));
1338  ...
1339 }
1340 
1341 TEST_P(FooTest, HasBlahBlah) {
1342  ...
1343 }
1344 ```
1345 
1346 Finally, you can use `INSTANTIATE_TEST_SUITE_P` to instantiate the test suite
1347 with any set of parameters you want. googletest defines a number of functions
1348 for generating test parameters. They return what we call (surprise!) *parameter
1349 generators*. Here is a summary of them, which are all in the `testing`
1350 namespace:
1351 
1352 <!-- mdformat off(github rendering does not support multiline tables) -->
1353 
1354 | Parameter Generator | Behavior |
1355 | ----------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------- |
1356 | `Range(begin, end [, step])` | Yields values `{begin, begin+step, begin+step+step, ...}`. The values do not include `end`. `step` defaults to 1. |
1357 | `Values(v1, v2, ..., vN)` | Yields values `{v1, v2, ..., vN}`. |
1358 | `ValuesIn(container)` and `ValuesIn(begin,end)` | Yields values from a C-style array, an STL-style container, or an iterator range `[begin, end)` |
1359 | `Bool()` | Yields sequence `{false, true}`. |
1360 | `Combine(g1, g2, ..., gN)` | Yields all combinations (Cartesian product) as std\:\:tuples of the values generated by the `N` generators. |
1361 
1362 <!-- mdformat on-->
1363 
1364 For more details, see the comments at the definitions of these functions.
1365 
1366 The following statement will instantiate tests from the `FooTest` test suite
1367 each with parameter values `"meeny"`, `"miny"`, and `"moe"`.
1368 
1369 ```c++
1370 INSTANTIATE_TEST_SUITE_P(InstantiationName,
1371  FooTest,
1372  testing::Values("meeny", "miny", "moe"));
1373 ```
1374 
1375 NOTE: The code above must be placed at global or namespace scope, not at
1376 function scope.
1377 
1378 NOTE: Don't forget this step! If you do your test will silently pass, but none
1379 of its suites will ever run!
1380 
1381 There is work in progress to make omitting `INSTANTIATE_TEST_SUITE_P` show up
1382 under the `GoogleTestVerification` test suite and to then make that an error.
1383 If you have a test suite where that omission is not an error, for example it is
1384 in a library that may be linked in for other reason or where the list of test
1385 cases is dynamic and may be empty, then this check can be suppressed by tagging
1386 the test suite:
1387 
1388 ```c++
1389 GTEST_ALLOW_UNINSTANTIATED_PARAMETERIZED_TEST(FooTest);
1390 ```
1391 
1392 To distinguish different instances of the pattern (yes, you can instantiate it
1393 more than once), the first argument to `INSTANTIATE_TEST_SUITE_P` is a prefix
1394 that will be added to the actual test suite name. Remember to pick unique
1395 prefixes for different instantiations. The tests from the instantiation above
1396 will have these names:
1397 
1398 * `InstantiationName/FooTest.DoesBlah/0` for `"meeny"`
1399 * `InstantiationName/FooTest.DoesBlah/1` for `"miny"`
1400 * `InstantiationName/FooTest.DoesBlah/2` for `"moe"`
1401 * `InstantiationName/FooTest.HasBlahBlah/0` for `"meeny"`
1402 * `InstantiationName/FooTest.HasBlahBlah/1` for `"miny"`
1403 * `InstantiationName/FooTest.HasBlahBlah/2` for `"moe"`
1404 
1405 You can use these names in [`--gtest_filter`](#running-a-subset-of-the-tests).
1406 
1407 This statement will instantiate all tests from `FooTest` again, each with
1408 parameter values `"cat"` and `"dog"`:
1409 
1410 ```c++
1411 const char* pets[] = {"cat", "dog"};
1412 INSTANTIATE_TEST_SUITE_P(AnotherInstantiationName, FooTest,
1413  testing::ValuesIn(pets));
1414 ```
1415 
1416 The tests from the instantiation above will have these names:
1417 
1418 * `AnotherInstantiationName/FooTest.DoesBlah/0` for `"cat"`
1419 * `AnotherInstantiationName/FooTest.DoesBlah/1` for `"dog"`
1420 * `AnotherInstantiationName/FooTest.HasBlahBlah/0` for `"cat"`
1421 * `AnotherInstantiationName/FooTest.HasBlahBlah/1` for `"dog"`
1422 
1423 Please note that `INSTANTIATE_TEST_SUITE_P` will instantiate *all* tests in the
1424 given test suite, whether their definitions come before or *after* the
1425 `INSTANTIATE_TEST_SUITE_P` statement.
1426 
1427 You can see [sample7_unittest.cc] and [sample8_unittest.cc] for more examples.

1432 ### Creating Value-Parameterized Abstract Tests
1433 
1434 In the above, we define and instantiate `FooTest` in the *same* source file.
1435 Sometimes you may want to define value-parameterized tests in a library and let
1436 other people instantiate them later. This pattern is known as *abstract tests*.
1437 As an example of its application, when you are designing an interface you can
1438 write a standard suite of abstract tests (perhaps using a factory function as
1439 the test parameter) that all implementations of the interface are expected to
1440 pass. When someone implements the interface, they can instantiate your suite to
1441 get all the interface-conformance tests for free.
1442 
1443 To define abstract tests, you should organize your code like this:
1444 
1445 1. Put the definition of the parameterized test fixture class (e.g. `FooTest`)
1446  in a header file, say `foo_param_test.h`. Think of this as *declaring* your
1447  abstract tests.
1448 2. Put the `TEST_P` definitions in `foo_param_test.cc`, which includes
1449  `foo_param_test.h`. Think of this as *implementing* your abstract tests.
1450 
1451 Once they are defined, you can instantiate them by including `foo_param_test.h`,
1452 invoking `INSTANTIATE_TEST_SUITE_P()`, and depending on the library target that
1453 contains `foo_param_test.cc`. You can instantiate the same abstract test suite
1454 multiple times, possibly in different source files.
1455 
1456 ### Specifying Names for Value-Parameterized Test Parameters
1457 
1458 The optional last argument to `INSTANTIATE_TEST_SUITE_P()` allows the user to
1459 specify a function or functor that generates custom test name suffixes based on
1460 the test parameters. The function should accept one argument of type
1461 `testing::TestParamInfo<class ParamType>`, and return `std::string`.
1462 
1463 `testing::PrintToStringParamName` is a builtin test suffix generator that
1464 returns the value of `testing::PrintToString(GetParam())`. It does not work for
1465 `std::string` or C strings.
1466 
1467 NOTE: test names must be non-empty, unique, and may only contain ASCII
1468 alphanumeric characters. In particular, they
1469 [should not contain underscores](faq.md#why-should-test-suite-names-and-test-names-not-contain-underscore)
1470 
1471 ```c++
1472 class MyTestSuite : public testing::TestWithParam<int> {};
1473 
1474 TEST_P(MyTestSuite, MyTest)
1475 {
1476  std::cout << "Example Test Param: " << GetParam() << std::endl;
1477 }
1478 
1479 INSTANTIATE_TEST_SUITE_P(MyGroup, MyTestSuite, testing::Range(0, 10),
1480  testing::PrintToStringParamName());
1481 ```
1482 
1483 Providing a custom functor allows for more control over test parameter name
1484 generation, especially for types where the automatic conversion does not
1485 generate helpful parameter names (e.g. strings as demonstrated above). The
1486 following example illustrates this for multiple parameters, an enumeration type
1487 and a string, and also demonstrates how to combine generators. It uses a lambda
1488 for conciseness:
1489 
1490 ```c++
1491 enum class MyType { MY_FOO = 0, MY_BAR = 1 };
1492 
1493 class MyTestSuite : public testing::TestWithParam<std::tuple<MyType, string>> {
1494 };
1495 
1496 INSTANTIATE_TEST_SUITE_P(
1497  MyGroup, MyTestSuite,
1498  testing::Combine(
1499  testing::Values(MyType::VALUE_0, MyType::VALUE_1),
1500  testing::ValuesIn("", "")),
1501  [](const testing::TestParamInfo<MyTestSuite::ParamType>& info) {
1502  string name = absl::StrCat(
1503  std::get<0>(info.param) == MY_FOO ? "Foo" : "Bar", "_",
1504  std::get<1>(info.param));
1505  absl::c_replace_if(name, [](char c) { return !std::isalnum(c); }, '_');
1506  return name;
1507  });
1508 ```

1510 ## Typed Tests
1511 
1512 Suppose you have multiple implementations of the same interface and want to make
1513 sure that all of them satisfy some common requirements. Or, you may have defined
1514 several types that are supposed to conform to the same "concept" and you want to
1515 verify it. In both cases, you want the same test logic repeated for different
1516 types.
1517 
1518 While you can write one `TEST` or `TEST_F` for each type you want to test (and
1519 you may even factor the test logic into a function template that you invoke from
1520 the `TEST`), it's tedious and doesn't scale: if you want `m` tests over `n`
1521 types, you'll end up writing `m*n` `TEST`s.
1522 
1523 *Typed tests* allow you to repeat the same test logic over a list of types. You
1524 only need to write the test logic once, although you must know the type list
1525 when writing typed tests. Here's how you do it:
1526 
1527 First, define a fixture class template. It should be parameterized by a type.
1528 Remember to derive it from `::testing::Test`:
1529 
1530 ```c++
1531 template <typename T>
1532 class FooTest : public ::testing::Test {
1533  public:
1534  ...
1535  typedef std::list<T> List;
1536  static T shared_;
1537  T value_;
1538 };
1539 ```
1540 
1541 Next, associate a list of types with the test suite, which will be repeated for
1542 each type in the list:
1543 
1544 ```c++
1545 using MyTypes = ::testing::Types<char, int, unsigned int>;
1546 TYPED_TEST_SUITE(FooTest, MyTypes);
1547 ```
1548 
1549 The type alias (`using` or `typedef`) is necessary for the `TYPED_TEST_SUITE`
1550 macro to parse correctly. Otherwise the compiler will think that each comma in
1551 the type list introduces a new macro argument.
1552 
1553 Then, use `TYPED_TEST()` instead of `TEST_F()` to define a typed test for this
1554 test suite. You can repeat this as many times as you want:
1555 
1556 ```c++
1557 TYPED_TEST(FooTest, DoesBlah) {
1558  // Inside a test, refer to the special name TypeParam to get the type
1559  // parameter. Since we are inside a derived class template, C++ requires
1560  // us to visit the members of FooTest via 'this'.
1561  TypeParam n = this->value_;
1562 
1563  // To visit static members of the fixture, add the 'TestFixture::'
1564  // prefix.
1565  n += TestFixture::shared_;
1566 
1567  // To refer to typedefs in the fixture, add the 'typename TestFixture::'
1568  // prefix. The 'typename' is required to satisfy the compiler.
1569  typename TestFixture::List values;
1570 
1571  values.push_back(n);
1572  ...
1573 }
1574 
1575 TYPED_TEST(FooTest, HasPropertyA) { ... }
1576 ```
1577 
1578 You can see [sample6_unittest.cc] for a complete example.
1579 
1580 [sample6_unittest.cc]: ../samples/sample6_unittest.cc "Typed Test example"

1582 ## Type-Parameterized Tests
1583 
1584 *Type-parameterized tests* are like typed tests, except that they don't require
1585 you to know the list of types ahead of time. Instead, you can define the test
1586 logic first and instantiate it with different type lists later. You can even
1587 instantiate it more than once in the same program.
1588 
1589 If you are designing an interface or concept, you can define a suite of
1590 type-parameterized tests to verify properties that any valid implementation of
1591 the interface/concept should have. Then, the author of each implementation can
1592 just instantiate the test suite with their type to verify that it conforms to
1593 the requirements, without having to write similar tests repeatedly. Here's an
1594 example:
1595 
1596 First, define a fixture class template, as we did with typed tests:
1597 
1598 ```c++
1599 template <typename T>
1600 class FooTest : public ::testing::Test {
1601  ...
1602 };
1603 ```
1604 
1605 Next, declare that you will define a type-parameterized test suite:
1606 
1607 ```c++
1608 TYPED_TEST_SUITE_P(FooTest);
1609 ```
1610 
1611 Then, use `TYPED_TEST_P()` to define a type-parameterized test. You can repeat
1612 this as many times as you want:
1613 
1614 ```c++
1615 TYPED_TEST_P(FooTest, DoesBlah) {
1616  // Inside a test, refer to TypeParam to get the type parameter.
1617  TypeParam n = 0;
1618  ...
1619 }
1620 
1621 TYPED_TEST_P(FooTest, HasPropertyA) { ... }
1622 ```
1623 
1624 Now the tricky part: you need to register all test patterns using the
1625 `REGISTER_TYPED_TEST_SUITE_P` macro before you can instantiate them. The first
1626 argument of the macro is the test suite name; the rest are the names of the
1627 tests in this test suite:
1628 
1629 ```c++
1630 REGISTER_TYPED_TEST_SUITE_P(FooTest,
1631  DoesBlah, HasPropertyA);
1632 ```
1633 
1634 Finally, you are free to instantiate the pattern with the types you want. If you
1635 put the above code in a header file, you can `#include` it in multiple C++
1636 source files and instantiate it multiple times.
1637 
1638 ```c++
1639 typedef ::testing::Types<char, int, unsigned int> MyTypes;
1640 INSTANTIATE_TYPED_TEST_SUITE_P(My, FooTest, MyTypes);
1641 ```
1642 
1643 To distinguish different instances of the pattern, the first argument to the
1644 `INSTANTIATE_TYPED_TEST_SUITE_P` macro is a prefix that will be added to the
1645 actual test suite name. Remember to pick unique prefixes for different
1646 instances.
1647 
1648 In the special case where the type list contains only one type, you can write
1649 that type directly without `::testing::Types<...>`, like this:
1650 
1651 ```c++
1652 INSTANTIATE_TYPED_TEST_SUITE_P(My, FooTest, int);
1653 ```
1654 
1655 You can see [sample6_unittest.cc] for a complete example.

1657 ## Testing Private Code
1658 
1659 If you change your software's internal implementation, your tests should not
1660 break as long as the change is not observable by users. Therefore, **per the
1661 black-box testing principle, most of the time you should test your code through
1662 its public interfaces.**
1663 
1664 **If you still find yourself needing to test internal implementation code,
1665 consider if there's a better design.** The desire to test internal
1666 implementation is often a sign that the class is doing too much. Consider
1667 extracting an implementation class, and testing it. Then use that implementation
1668 class in the original class.
1669 
1670 If you absolutely have to test non-public interface code though, you can. There
1671 are two cases to consider:
1672 
1673 * Static functions ( *not* the same as static member functions!) or unnamed
1674  namespaces, and
1675 * Private or protected class members
1676 
1677 To test them, we use the following special techniques:
1678 
1679 * Both static functions and definitions/declarations in an unnamed namespace
1680  are only visible within the same translation unit. To test them, you can
1681  `#include` the entire `.cc` file being tested in your `*_test.cc` file.
1682  (#including `.cc` files is not a good way to reuse code - you should not do
1683  this in production code!)
1684 
1685  However, a better approach is to move the private code into the
1686  `foo::internal` namespace, where `foo` is the namespace your project
1687  normally uses, and put the private declarations in a `*-internal.h` file.
1688  Your production `.cc` files and your tests are allowed to include this
1689  internal header, but your clients are not. This way, you can fully test your
1690  internal implementation without leaking it to your clients.
1691 
1692 * Private class members are only accessible from within the class or by
1693  friends. To access a class' private members, you can declare your test
1694  fixture as a friend to the class and define accessors in your fixture. Tests
1695  using the fixture can then access the private members of your production
1696  class via the accessors in the fixture. Note that even though your fixture
1697  is a friend to your production class, your tests are not automatically
1698  friends to it, as they are technically defined in sub-classes of the
1699  fixture.
1700 
1701  Another way to test private members is to refactor them into an
1702  implementation class, which is then declared in a `*-internal.h` file. Your
1703  clients aren't allowed to include this header but your tests can. Such is
1704  called the
1705  [Pimpl](https://www.gamedev.net/articles/programming/general-and-gameplay-programming/the-c-pimpl-r1794/)
1706  (Private Implementation) idiom.
1707 
1708  Or, you can declare an individual test as a friend of your class by adding
1709  this line in the class body:
1710 
1711  ```c++
1712  FRIEND_TEST(TestSuiteName, TestName);
1713  ```
1714 
1715  For example,
1716 
1717  ```c++
1718  // foo.h
1719  class Foo {
1720  ...
1721  private:
1722  FRIEND_TEST(FooTest, BarReturnsZeroOnNull);
1723 
1724  int Bar(void* x);
1725  };
1726 
1727  // foo_test.cc
1728  ...
1729  TEST(FooTest, BarReturnsZeroOnNull) {
1730  Foo foo;
1731  EXPECT_EQ(foo.Bar(NULL), 0); // Uses Foo's private member Bar().
1732  }
1733  ```
1734 
1735  Pay special attention when your class is defined in a namespace, as you
1736  should define your test fixtures and tests in the same namespace if you want
1737  them to be friends of your class. For example, if the code to be tested
1738  looks like:
1739 
1740  ```c++
1741  namespace my_namespace {
1742 
1743  class Foo {
1744  friend class FooTest;
1745  FRIEND_TEST(FooTest, Bar);
1746  FRIEND_TEST(FooTest, Baz);
1747  ... definition of the class Foo ...
1748  };
1749 
1750  } // namespace my_namespace
1751  ```
1752 
1753  Your test code should be something like:
1754 
1755  ```c++
1756  namespace my_namespace {
1757 
1758  class FooTest : public ::testing::Test {
1759  protected:
1760  ...
1761  };
1762 
1763  TEST_F(FooTest, Bar) { ... }
1764  TEST_F(FooTest, Baz) { ... }
1765 
1766  } // namespace my_namespace
1767  ```

1769 ## "Catching" Failures
1770 
1771 If you are building a testing utility on top of googletest, you'll want to test
1772 your utility. What framework would you use to test it? googletest, of course.
1773 
1774 The challenge is to verify that your testing utility reports failures correctly.
1775 In frameworks that report a failure by throwing an exception, you could catch
1776 the exception and assert on it. But googletest doesn't use exceptions, so how do
1777 we test that a piece of code generates an expected failure?
1778 
1779 `"gtest/gtest-spi.h"` contains some constructs to do this. After #including this header,
1780 you can use
1781 
1782 ```c++
1783  EXPECT_FATAL_FAILURE(statement, substring);
1784 ```
1785 
1786 to assert that `statement` generates a fatal (e.g. `ASSERT_*`) failure in the
1787 current thread whose message contains the given `substring`, or use
1788 
1789 ```c++
1790  EXPECT_NONFATAL_FAILURE(statement, substring);
1791 ```
1792 
1793 if you are expecting a non-fatal (e.g. `EXPECT_*`) failure.
1794 
1795 Only failures in the current thread are checked to determine the result of this
1796 type of expectations. If `statement` creates new threads, failures in these
1797 threads are also ignored. If you want to catch failures in other threads as
1798 well, use one of the following macros instead:
1799 
1800 ```c++
1801  EXPECT_FATAL_FAILURE_ON_ALL_THREADS(statement, substring);
1802  EXPECT_NONFATAL_FAILURE_ON_ALL_THREADS(statement, substring);
1803 ```
1804 
1805 NOTE: Assertions from multiple threads are currently not supported on Windows.
1806 
1807 For technical reasons, there are some caveats:
1808 
1809 1. You cannot stream a failure message to either macro.
1810 
1811 2. `statement` in `EXPECT_FATAL_FAILURE{_ON_ALL_THREADS}()` cannot reference
1812  local non-static variables or non-static members of `this` object.
1813 
1814 3. `statement` in `EXPECT_FATAL_FAILURE{_ON_ALL_THREADS}()` cannot return a
1815  value.
1816 
1817 ## Registering tests programmatically
1818 
1819 The `TEST` macros handle the vast majority of all use cases, but there are few
1820 where runtime registration logic is required. For those cases, the framework
1821 provides the `::testing::RegisterTest` that allows callers to register arbitrary
1822 tests dynamically.
1823 
1824 This is an advanced API only to be used when the `TEST` macros are insufficient.
1825 The macros should be preferred when possible, as they avoid most of the
1826 complexity of calling this function.
1827 
1828 It provides the following signature:
1829 
1830 ```c++
1831 template <typename Factory>
1832 TestInfo* RegisterTest(const char* test_suite_name, const char* test_name,
1833  const char* type_param, const char* value_param,
1834  const char* file, int line, Factory factory);
1835 ```
1836 
1837 The `factory` argument is a factory callable (move-constructible) object or
1838 function pointer that creates a new instance of the Test object. It handles
1839 ownership to the caller. The signature of the callable is `Fixture*()`, where
1840 `Fixture` is the test fixture class for the test. All tests registered with the
1841 same `test_suite_name` must return the same fixture type. This is checked at
1842 runtime.
1843 
1844 The framework will infer the fixture class from the factory and will call the
1845 `SetUpTestSuite` and `TearDownTestSuite` for it.
1846 
1847 Must be called before `RUN_ALL_TESTS()` is invoked, otherwise behavior is
1848 undefined.
1849 
1850 Use case example:
1851 
1852 ```c++
1853 class MyFixture : public ::testing::Test {
1854  public:
1855  // All of these optional, just like in regular macro usage.
1856  static void SetUpTestSuite() { ... }
1857  static void TearDownTestSuite() { ... }
1858  void SetUp() override { ... }
1859  void TearDown() override { ... }
1860 };
1861 
1862 class MyTest : public MyFixture {
1863  public:
1864  explicit MyTest(int data) : data_(data) {}
1865  void TestBody() override { ... }
1866 
1867  private:
1868  int data_;
1869 };
1870 
1871 void RegisterMyTests(const std::vector<int>& values) {
1872  for (int v : values) {
1873  ::testing::RegisterTest(
1874  "MyFixture", ("Test" + std::to_string(v)).c_str(), nullptr,
1875  std::to_string(v).c_str(),
1876  __FILE__, __LINE__,
1877  // Important to use the fixture type as the return type here.
1878  [=]() -> MyFixture* { return new MyTest(v); });
1879  }
1880 }
1881 ...
1882 int main(int argc, char** argv) {
1883  std::vector<int> values_to_test = LoadValuesFromConfig();
1884  RegisterMyTests(values_to_test);
1885  ...
1886  return RUN_ALL_TESTS();
1887 }
1888 ```
1889 ## Getting the Current Test's Name
1890 
1891 Sometimes a function may need to know the name of the currently running test.
1892 For example, you may be using the `SetUp()` method of your test fixture to set
1893 the golden file name based on which test is running. The `::testing::TestInfo`
1894 class has this information:
1895 
1896 ```c++
1897 namespace testing {
1898 
1899 class TestInfo {
1900  public:
1901  // Returns the test suite name and the test name, respectively.
1902  //
1903  // Do NOT delete or free the return value - it's managed by the
1904  // TestInfo class.
1905  const char* test_suite_name() const;
1906  const char* name() const;
1907 };
1908 
1909 }
1910 ```
1911 
1912 To obtain a `TestInfo` object for the currently running test, call
1913 `current_test_info()` on the `UnitTest` singleton object:
1914 
1915 ```c++
1916  // Gets information about the currently running test.
1917  // Do NOT delete the returned object - it's managed by the UnitTest class.
1918  const ::testing::TestInfo* const test_info =
1919  ::testing::UnitTest::GetInstance()->current_test_info();
1920 
1921  printf("We are in test %s of test suite %s.\n",
1922  test_info->name(),
1923  test_info->test_suite_name());
1924 ```
1925 
1926 `current_test_info()` returns a null pointer if no test is running. In
1927 particular, you cannot find the test suite name in `SetUpTestSuite()`,
1928 `TearDownTestSuite()` (where you know the test suite name implicitly), or
1929 functions called from them.

1931 ## Extending googletest by Handling Test Events
1932 
1933 googletest provides an **event listener API** to let you receive notifications
1934 about the progress of a test program and test failures. The events you can
1935 listen to include the start and end of the test program, a test suite, or a test
1936 method, among others. You may use this API to augment or replace the standard
1937 console output, replace the XML output, or provide a completely different form
1938 of output, such as a GUI or a database. You can also use test events as
1939 checkpoints to implement a resource leak checker, for example.
1940 
1941 ### Defining Event Listeners
1942 
1943 To define a event listener, you subclass either testing::TestEventListener or
1944 testing::EmptyTestEventListener The former is an (abstract) interface, where
1945 *each pure virtual method can be overridden to handle a test event* (For
1946 example, when a test starts, the `OnTestStart()` method will be called.). The
1947 latter provides an empty implementation of all methods in the interface, such
1948 that a subclass only needs to override the methods it cares about.
1949 
1950 When an event is fired, its context is passed to the handler function as an
1951 argument. The following argument types are used:
1952 
1953 * UnitTest reflects the state of the entire test program,
1954 * TestSuite has information about a test suite, which can contain one or more
1955  tests,
1956 * TestInfo contains the state of a test, and
1957 * TestPartResult represents the result of a test assertion.
1958 
1959 An event handler function can examine the argument it receives to find out
1960 interesting information about the event and the test program's state.
1961 
1962 Here's an example:
1963 
1964 ```c++
1965  class MinimalistPrinter : public ::testing::EmptyTestEventListener {
1966  // Called before a test starts.
1967  virtual void OnTestStart(const ::testing::TestInfo& test_info) {
1968  printf("*** Test %s.%s starting.\n",
1969  test_info.test_suite_name(), test_info.name());
1970  }
1971 
1972  // Called after a failed assertion or a SUCCESS().
1973  virtual void OnTestPartResult(const ::testing::TestPartResult& test_part_result) {
1974  printf("%s in %s:%d\n%s\n",
1975  test_part_result.failed() ? "*** Failure" : "Success",
1976  test_part_result.file_name(),
1977  test_part_result.line_number(),
1978  test_part_result.summary());
1979  }
1980 
1981  // Called after a test ends.
1982  virtual void OnTestEnd(const ::testing::TestInfo& test_info) {
1983  printf("*** Test %s.%s ending.\n",
1984  test_info.test_suite_name(), test_info.name());
1985  }
1986  };
1987 ```
1988 
1989 ### Using Event Listeners
1990 
1991 To use the event listener you have defined, add an instance of it to the
1992 googletest event listener list (represented by class TestEventListeners - note
1993 the "s" at the end of the name) in your `main()` function, before calling
1994 `RUN_ALL_TESTS()`:
1995 
1996 ```c++
1997 int main(int argc, char** argv) {
1998  ::testing::InitGoogleTest(&argc, argv);
1999  // Gets hold of the event listener list.
2000  ::testing::TestEventListeners& listeners =
2001  ::testing::UnitTest::GetInstance()->listeners();
2002  // Adds a listener to the end. googletest takes the ownership.
2003  listeners.Append(new MinimalistPrinter);
2004  return RUN_ALL_TESTS();
2005 }
2006 ```
2007 
2008 There's only one problem: the default test result printer is still in effect, so
2009 its output will mingle with the output from your minimalist printer. To suppress
2010 the default printer, just release it from the event listener list and delete it.
2011 You can do so by adding one line:
2012 
2013 ```c++
2014  ...
2015  delete listeners.Release(listeners.default_result_printer());
2016  listeners.Append(new MinimalistPrinter);
2017  return RUN_ALL_TESTS();
2018 ```
2019 
2020 Now, sit back and enjoy a completely different output from your tests. For more
2021 details, see [sample9_unittest.cc].
2022 
2023 [sample9_unittest.cc]: ../samples/sample9_unittest.cc "Event listener example"
2024 
2025 You may append more than one listener to the list. When an `On*Start()` or
2026 `OnTestPartResult()` event is fired, the listeners will receive it in the order
2027 they appear in the list (since new listeners are added to the end of the list,
2028 the default text printer and the default XML generator will receive the event
2029 first). An `On*End()` event will be received by the listeners in the *reverse*
2030 order. This allows output by listeners added later to be framed by output from
2031 listeners added earlier.
2032 
2033 ### Generating Failures in Listeners
2034 
2035 You may use failure-raising macros (`EXPECT_*()`, `ASSERT_*()`, `FAIL()`, etc)
2036 when processing an event. There are some restrictions:
2037 
2038 1. You cannot generate any failure in `OnTestPartResult()` (otherwise it will
2039  cause `OnTestPartResult()` to be called recursively).
2040 2. A listener that handles `OnTestPartResult()` is not allowed to generate any
2041  failure.
2042 
2043 When you add listeners to the listener list, you should put listeners that
2044 handle `OnTestPartResult()` *before* listeners that can generate failures. This
2045 ensures that failures generated by the latter are attributed to the right test
2046 by the former.
2047 
2048 See [sample10_unittest.cc] for an example of a failure-raising listener.
2049

2052 ## Running Test Programs: Advanced Options
2053 
2054 googletest test programs are ordinary executables. Once built, you can run them
2055 directly and affect their behavior via the following environment variables
2056 and/or command line flags. For the flags to work, your programs must call
2057 `::testing::InitGoogleTest()` before calling `RUN_ALL_TESTS()`.
2058 
2059 To see a list of supported flags and their usage, please run your test program
2060 with the `--help` flag. You can also use `-h`, `-?`, or `/?` for short.
2061 
2062 If an option is specified both by an environment variable and by a flag, the
2063 latter takes precedence.
2064 
2065 ### Selecting Tests
2066 
2067 #### Listing Test Names
2068 
2069 Sometimes it is necessary to list the available tests in a program before
2070 running them so that a filter may be applied if needed. Including the flag
2071 `--gtest_list_tests` overrides all other flags and lists tests in the following
2072 format:
2073 
2074 ```none
2075 TestSuite1.
2076  TestName1
2077  TestName2
2078 TestSuite2.
2079  TestName
2080 ```
2081 
2082 None of the tests listed are actually run if the flag is provided. There is no
2083 corresponding environment variable for this flag.
2084 
2085 #### Running a Subset of the Tests
2086 
2087 By default, a googletest program runs all tests the user has defined. Sometimes,
2088 you want to run only a subset of the tests (e.g. for debugging or quickly
2089 verifying a change). If you set the `GTEST_FILTER` environment variable or the
2090 `--gtest_filter` flag to a filter string, googletest will only run the tests
2091 whose full names (in the form of `TestSuiteName.TestName`) match the filter.
2092 
2093 The format of a filter is a '`:`'-separated list of wildcard patterns (called
2094 the *positive patterns*) optionally followed by a '`-`' and another
2095 '`:`'-separated pattern list (called the *negative patterns*). A test matches
2096 the filter if and only if it matches any of the positive patterns but does not
2097 match any of the negative patterns.
2098 
2099 A pattern may contain `'*'` (matches any string) or `'?'` (matches any single
2100 character). For convenience, the filter `'*-NegativePatterns'` can be also
2101 written as `'-NegativePatterns'`.
2102 
2103 For example:
2104 
2105 * `./foo_test` Has no flag, and thus runs all its tests.
2106 * `./foo_test --gtest_filter=*` Also runs everything, due to the single
2107  match-everything `*` value.
2108 * `./foo_test --gtest_filter=FooTest.*` Runs everything in test suite
2109  `FooTest` .
2110 * `./foo_test --gtest_filter=*Null*:*Constructor*` Runs any test whose full
2111  name contains either `"Null"` or `"Constructor"` .
2112 * `./foo_test --gtest_filter=-*DeathTest.*` Runs all non-death tests.
2113 * `./foo_test --gtest_filter=FooTest.*-FooTest.Bar` Runs everything in test
2114  suite `FooTest` except `FooTest.Bar`.
2115 * `./foo_test --gtest_filter=FooTest.*:BarTest.*-FooTest.Bar:BarTest.Foo` Runs
2116  everything in test suite `FooTest` except `FooTest.Bar` and everything in
2117  test suite `BarTest` except `BarTest.Foo`.
2118 
2119 #### Stop test execution upon first failure
2120 
2121 By default, a googletest program runs all tests the user has defined. In some
2122 cases (e.g. iterative test development & execution) it may be desirable stop
2123 test execution upon first failure (trading improved latency for completeness).
2124 If `GTEST_FAIL_FAST` environment variable or `--gtest_fail_fast` flag is set,
2125 the test runner will stop execution as soon as the first test failure is
2126 found.
2127 
2128 #### Temporarily Disabling Tests
2129 
2130 If you have a broken test that you cannot fix right away, you can add the
2131 `DISABLED_` prefix to its name. This will exclude it from execution. This is
2132 better than commenting out the code or using `#if 0`, as disabled tests are
2133 still compiled (and thus won't rot).
2134 
2135 If you need to disable all tests in a test suite, you can either add `DISABLED_`
2136 to the front of the name of each test, or alternatively add it to the front of
2137 the test suite name.
2138 
2139 For example, the following tests won't be run by googletest, even though they
2140 will still be compiled:
2141 
2142 ```c++
2143 // Tests that Foo does Abc.
2144 TEST(FooTest, DISABLED_DoesAbc) { ... }
2145 
2146 class DISABLED_BarTest : public ::testing::Test { ... };
2147 
2148 // Tests that Bar does Xyz.
2149 TEST_F(DISABLED_BarTest, DoesXyz) { ... }
2150 ```
2151 
2152 NOTE: This feature should only be used for temporary pain-relief. You still have
2153 to fix the disabled tests at a later date. As a reminder, googletest will print
2154 a banner warning you if a test program contains any disabled tests.
2155 
2156 TIP: You can easily count the number of disabled tests you have using `gsearch`
2157 and/or `grep`. This number can be used as a metric for improving your test
2158 quality.
2159 
2160 #### Temporarily Enabling Disabled Tests
2161 
2162 To include disabled tests in test execution, just invoke the test program with
2163 the `--gtest_also_run_disabled_tests` flag or set the
2164 `GTEST_ALSO_RUN_DISABLED_TESTS` environment variable to a value other than `0`.
2165 You can combine this with the `--gtest_filter` flag to further select which
2166 disabled tests to run.
2167 
2168 ### Repeating the Tests
2169 
2170 Once in a while you'll run into a test whose result is hit-or-miss. Perhaps it
2171 will fail only 1% of the time, making it rather hard to reproduce the bug under
2172 a debugger. This can be a major source of frustration.
2173 
2174 The `--gtest_repeat` flag allows you to repeat all (or selected) test methods in
2175 a program many times. Hopefully, a flaky test will eventually fail and give you
2176 a chance to debug. Here's how to use it:
2177 
2178 ```none
2179 $ foo_test --gtest_repeat=1000
2180 Repeat foo_test 1000 times and don't stop at failures.
2181 
2182 $ foo_test --gtest_repeat=-1
2183 A negative count means repeating forever.
2184 
2185 $ foo_test --gtest_repeat=1000 --gtest_break_on_failure
2186 Repeat foo_test 1000 times, stopping at the first failure. This
2187 is especially useful when running under a debugger: when the test
2188 fails, it will drop into the debugger and you can then inspect
2189 variables and stacks.
2190 
2191 $ foo_test --gtest_repeat=1000 --gtest_filter=FooBar.*
2192 Repeat the tests whose name matches the filter 1000 times.
2193 ```
2194 
2195 If your test program contains
2196 [global set-up/tear-down](#global-set-up-and-tear-down) code, it will be
2197 repeated in each iteration as well, as the flakiness may be in it. You can also
2198 specify the repeat count by setting the `GTEST_REPEAT` environment variable.
2199 
2200 ### Shuffling the Tests
2201 
2202 You can specify the `--gtest_shuffle` flag (or set the `GTEST_SHUFFLE`
2203 environment variable to `1`) to run the tests in a program in a random order.
2204 This helps to reveal bad dependencies between tests.
2205 
2206 By default, googletest uses a random seed calculated from the current time.
2207 Therefore you'll get a different order every time. The console output includes
2208 the random seed value, such that you can reproduce an order-related test failure
2209 later. To specify the random seed explicitly, use the `--gtest_random_seed=SEED`
2210 flag (or set the `GTEST_RANDOM_SEED` environment variable), where `SEED` is an
2211 integer in the range [0, 99999]. The seed value 0 is special: it tells
2212 googletest to do the default behavior of calculating the seed from the current
2213 time.
2214 
2215 If you combine this with `--gtest_repeat=N`, googletest will pick a different
2216 random seed and re-shuffle the tests in each iteration.

2287 #### Generating an XML Report
2288 
2289 googletest can emit a detailed XML report to a file in addition to its normal
2290 textual output. The report contains the duration of each test, and thus can help
2291 you identify slow tests.
2292 
2293 To generate the XML report, set the `GTEST_OUTPUT` environment variable or the
2294 `--gtest_output` flag to the string `"xml:path_to_output_file"`, which will
2295 create the file at the given location. You can also just use the string `"xml"`,
2296 in which case the output can be found in the `test_detail.xml` file in the
2297 current directory.
2298 
2299 If you specify a directory (for example, `"xml:output/directory/"` on Linux or
2300 `"xml:output\directory\"` on Windows), googletest will create the XML file in
2301 that directory, named after the test executable (e.g. `foo_test.xml` for test
2302 program `foo_test` or `foo_test.exe`). If the file already exists (perhaps left
2303 over from a previous run), googletest will pick a different name (e.g.
2304 `foo_test_1.xml`) to avoid overwriting it.
2305 
2306 The report is based on the `junitreport` Ant task. Since that format was
2307 originally intended for Java, a little interpretation is required to make it
2308 apply to googletest tests, as shown here:
2309 
2310 ```xml
2311 <testsuites name="AllTests" ...>
2312  <testsuite name="test_case_name" ...>
2313  <testcase name="test_name" ...>
2314  <failure message="..."/>
2315  <failure message="..."/>
2316  <failure message="..."/>
2317  </testcase>
2318  </testsuite>
2319 </testsuites>
2320 ```
2321 
2322 * The root `<testsuites>` element corresponds to the entire test program.
2323 * `<testsuite>` elements correspond to googletest test suites.
2324 * `<testcase>` elements correspond to googletest test functions.
2325 
2326 For instance, the following program
2327 
2328 ```c++
2329 TEST(MathTest, Addition) { ... }
2330 TEST(MathTest, Subtraction) { ... }
2331 TEST(LogicTest, NonContradiction) { ... }
2332 ```
2333 
2334 could generate this report:
2335 
2336 ```xml
2337 <?xml version="1.0" encoding="UTF-8"?>
2338 <testsuites tests="3" failures="1" errors="0" time="0.035" timestamp="2011-10-31T18:52:42" name="AllTests">
2339  <testsuite name="MathTest" tests="2" failures="1" errors="0" time="0.015">
2340  <testcase name="Addition" status="run" time="0.007" classname="">
2341  <failure message="Value of: add(1, 1)&#x0A; Actual: 3&#x0A;Expected: 2" type="">...</failure>
2342  <failure message="Value of: add(1, -1)&#x0A; Actual: 1&#x0A;Expected: 0" type="">...</failure>
2343  </testcase>
2344  <testcase name="Subtraction" status="run" time="0.005" classname="">
2345  </testcase>
2346  </testsuite>
2347  <testsuite name="LogicTest" tests="1" failures="0" errors="0" time="0.005">
2348  <testcase name="NonContradiction" status="run" time="0.005" classname="">
2349  </testcase>
2350  </testsuite>
2351 </testsuites>
2352 ```
2353 
2354 Things to note:
2355 
2356 * The `tests` attribute of a `<testsuites>` or `<testsuite>` element tells how
2357  many test functions the googletest program or test suite contains, while the
2358  `failures` attribute tells how many of them failed.
2359 
2360 * The `time` attribute expresses the duration of the test, test suite, or
2361  entire test program in seconds.
2362 
2363 * The `timestamp` attribute records the local date and time of the test
2364  execution.
2365 
2366 * Each `<failure>` element corresponds to a single failed googletest
2367  assertion.

2568 ### Controlling How Failures Are Reported
2569 
2570 #### Detecting Test Premature Exit
2571 
2572 Google Test implements the _premature-exit-file_ protocol for test runners
2573 to catch any kind of unexpected exits of test programs. Upon start,
2574 Google Test creates the file which will be automatically deleted after
2575 all work has been finished. Then, the test runner can check if this file
2576 exists. In case the file remains undeleted, the inspected test has exited
2577 prematurely.
2578 
2579 This feature is enabled only if the `TEST_PREMATURE_EXIT_FILE` environment
2580 variable has been set.
2581 
2582 #### Turning Assertion Failures into Break-Points
2583 
2584 When running test programs under a debugger, it's very convenient if the
2585 debugger can catch an assertion failure and automatically drop into interactive
2586 mode. googletest's *break-on-failure* mode supports this behavior.
2587 
2588 To enable it, set the `GTEST_BREAK_ON_FAILURE` environment variable to a value
2589 other than `0`. Alternatively, you can use the `--gtest_break_on_failure`
2590 command line flag.
2591 
2592 #### Disabling Catching Test-Thrown Exceptions
2593 
2594 googletest can be used either with or without exceptions enabled. If a test
2595 throws a C++ exception or (on Windows) a structured exception (SEH), by default
2596 googletest catches it, reports it as a test failure, and continues with the next
2597 test method. This maximizes the coverage of a test run. Also, on Windows an
2598 uncaught exception will cause a pop-up window, so catching the exceptions allows
2599 you to run the tests automatically.
2600 
2601 When debugging the test failures, however, you may instead want the exceptions
2602 to be handled by the debugger, such that you can examine the call stack when an
2603 exception is thrown. To achieve that, set the `GTEST_CATCH_EXCEPTIONS`
2604 environment variable to `0`, or use the `--gtest_catch_exceptions=0` flag when
2605 running the tests.
